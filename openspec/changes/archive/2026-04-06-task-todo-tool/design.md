## Context

Today the model has no structured working memory for its own plan within a task execution. Session notes exist but are auto-generated by a background process every 5 turns — the model cannot write to them directly. When compaction fires, pending items land in section 7 ("Pending Tasks") of the compaction summary as lossy prose. Sub-agents launched via `spawn_agent` start completely fresh — they receive only the raw `instructions` string from the parent; no conversation history, no session notes.

The result: on long tasks the model re-investigates things it already solved, loses sub-task granularity across compaction, and can't hand off structured context to sub-agents.

## Goals / Non-Goals

**Goals:**
- Model-controlled, DB-persisted todo list scoped per task
- Todos survive context compaction and task restarts (re-injected fresh from DB)
- Minimal token cost: only id/title/status injected system-wide; `context` and `result` on demand
- Sub-agents can discover and consume todo context via tools
- Collapsible UI panel showing real-time progress above the chat input
- Compaction prompt updated to not re-summarize items already in the todo system

**Non-Goals:**
- Todos are not board-level tasks — they don't appear on the Kanban board
- No automatic delegation — the model decides when to spawn agents; todos are informational
- Todos don't cross task boundaries (strictly task-scoped)
- No todo ordering or priority fields in v1

## Decisions

### Individual tools over full-replace

Copilot uses a full-replace pattern (`manage_todo_list(completeList)`), which is convenient when the list is always in working context. In Railyin, todos persist across compaction — 20 turns after creation the model may not remember every item. A full-replace at that point silently drops items the model forgot. Individual tools (`create_todo`, `update_todo`, `delete_todo`) with stable IDs are safer: the model operates on what it knows and doesn't touch what it doesn't.

**Alternative considered:** Full-replace + always injecting the full list so the model always has the latest state. Rejected because the injection is the safety net, not the primary interface — relying on it to make full-replace safe conflates the two concerns.

### Inject id/title/status only — context and result are on-demand

The injection block runs on every API call. Injecting `context` and `result` for every todo would burn tokens continuously for information that's only needed at delegation time or at review time. The model calls `get_todo(id)` when it's about to act on a specific item — that's the natural read point. `context` is written once at creation and read rarely, making it a poor candidate for continuous injection.

### DB persistence — not in-memory

In-memory state (like Copilot's todo) dies with the run. Railyin tasks can be paused, restarted, and handed to sub-agents across separate engine runs. DB persistence is required. A lightweight `task_todos` table suffices — no foreign keys beyond `task_id`.

### Inject all todos regardless of status

The injected list (id/title/status) serves two purposes: avoid model re-creating duplicates, and give the model a "scoreboard" of what's done vs. pending. Completed todos with a ✓ marker are low-token and useful signal. The model is responsible for deleting todos that are no longer relevant — deletion is intentional cleanup, not automatic.

**Alternative considered:** Inject only `not-started` and `in-progress`. Rejected because the model may want to refer to completed items by ID (e.g. "I already handled this in todo #2") and filtering them from the injection would break `get_todo` references in the conversation.

### `result` field as structured handoff

`spawn_agent` already returns a result string that lands in the conversation history as a `tool_result` message — but this gets lost to compaction prose. A `result` field on the todo is the sub-agent's explicit, lossless handoff note, readable by the parent long after compaction via `get_todo(id)`. It's also writable by the parent model directly (post-review, post-inline-work) without requiring a sub-agent.

### Sub-agents access todos via tools, not injection

Sub-agents (`runSubExecution`) build their own minimal `AIMessage[]` — they don't receive system injections. Requiring sub-agents to call `list_todos()` then `get_todo(id)` is an explicit, intentional pattern: the sub-agent declares what context it's using. Injecting into sub-agents would require architectural changes to `runSubExecution` and would give every sub-agent todos irrelevant to their scope.

## Risks / Trade-offs

- **Model forgetting IDs after compaction** → Mitigated by the always-injected id/title/status list; the model can also call `list_todos()` to recover all IDs at any time.
- **Todos growing unbounded on very long tasks** → The model should be prompted (via tool descriptions) to delete completed todos when they're no longer referenced. No hard enforcement in v1.
- **compaction-prompt section 7 drift** → Mitigated by explicit instruction in the updated compaction prompt to defer to the todo system. Spec delta captures this.
- **Sub-agent must spend an extra tool call to get context** → Acceptable; it's one `get_todo(id)` call. The parent typically prepopulates the instructions string with the ID so the sub-agent knows exactly what to fetch.

## Migration Plan

1. DB migration: add `task_todos` table (`id`, `task_id`, `title`, `status`, `context`, `result`, `created_at`, `updated_at`)
2. Add `todos` tool group to `tools.ts` (no breaking changes to existing groups)
3. Add todo injection to `compactMessages()` in `engine.ts` — additive system block
4. Update `COMPACTION_SYSTEM_PROMPT` — additive instruction to section 7
5. Add RPC handler for UI reads (read-only, model writes via tools)
6. Add collapsible Vue component in task detail / chat view
7. No rollback complexity — all changes are additive; removing the injection block and tool group reverts the behavior

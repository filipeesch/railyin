## Why

The Bun backend stores the full conversation history (every chat turn, tool call, and tool result) in a single shared SQLite file. As conversations grow, this causes write contention (single-writer WAL) that manifests as UI slowness, and makes ad-hoc debugging of a single conversation harder than it should be (you have to query a shared multi-tenant database instead of inspecting one file). Two other high-frequency tables (`stream_events`, `model_raw_messages`) compound the write load. Moving durable conversation content to per-conversation files in the user's profile folder — and dropping the tables that turn out to be unnecessary — reduces shared-DB write volume, improves debuggability, and simplifies the storage model.

## What Changes

- **New `ConversationMessageStore` abstraction**: a single interface for reading/writing conversation messages, replacing ~10 call sites that currently issue raw SQL against `conversation_messages` directly.
- **New file-backed implementation** (`FileConversationMessageStore`): each conversation's durable message history is stored as an append-only JSON Lines file at `~/.railyn/conversations/<conversationId>.jsonl`, with a small sidecar `<conversationId>.meta.json` index (line count / next id, last `compaction_summary` id + byte offset, byte length) for O(1) hot-query support (compaction anchor lookup, point lookup by id, pagination) instead of full-file scans.
- **Legacy read path preserved**: pre-existing conversations already in SQLite are NOT migrated/backfilled. A `LegacySqliteConversationMessageStore` implementation continues to serve them read-only; a thin resolver picks the correct implementation per conversation. Only conversations created after this change land in files.
- **Message `id` becomes the 1-based line number** in the conversation's file — derived from file position, not stored redundantly.
- **Concurrency**: writes to a given conversation's file are serialized via an in-process async write queue (no OS-level file locking needed — single Bun process).
- **`stream_events` table and mechanism removed entirely** (**BREAKING** for the `conversations.getStreamEvents` RPC, which is deleted). Investigation confirmed this table's only purpose — WebSocket reconnect-replay of live stream blocks — has no live caller anywhere in the frontend today. Removing it eliminates an entire write-buffer/table/migration/RPC/enrichment subsystem with no loss of used functionality. Live per-token streaming continues unaffected: it is broadcast directly over the WebSocket from `StreamProcessor`, in-memory, and was never durably persisted by design.
- **`model_raw_messages` moves to a file-based debug log** (`~/.railyn/debug/<executionId>.jsonl` or similar), preserving its sole purpose — manual forensic inspection during debugging — without occupying the shared SQLite file. Retention policy (short-lived, e.g. 1 day) is preserved via file cleanup instead of `DELETE` statements.
- **Deletion hooks**: a standalone injected `ConversationFileDeleter` collaborator (mirroring the existing optional `worktreeManager` constructor param pattern) is wired into `tasks.delete`, the AI-invoked task-deletion tool (`board-tool-executor`), and `RetentionJob`'s existing 7-day archived-chat-session sweep, deleting the corresponding conversation file(s) and debug log files alongside their existing SQL deletes. No new `chatSessions.delete` RPC is introduced — chat session deletion continues to mean archive-then-7-day-sweep, and the file cleanup attaches to that existing sweep.
- **`session-memory.ts` is refactored, not just path-fixed**: `extractSessionMemory`/`_doExtract` currently call `getDb()` internally and read `conversation_messages` directly — left as-is, this would silently stop seeing recent messages for file-backed conversations. It now accepts an injected `ConversationMessageStore` like every other call site, and gains its first-ever unit tests. The inconsistent `~/.config/railyn/tasks/<id>/session-notes.md` path is also aligned with the `~/.railyn` data-dir convention used everywhere else.
- **Testing strategy**: a shared `ConversationMessageStore` contract test suite runs identical cases against both implementations; write-queue concurrency, sidecar drift/self-heal, and tombstone handling get dedicated unit tests using real temp directories (no new filesystem abstraction introduced solely for mocking); `e2e/api` gains a pagination-parity test across both storage media; Playwright test changes are limited to removing the now-stale `getStreamEvents` mock default and a misleading comment — no new UI specs needed since RPC mock shapes are unchanged.

## Capabilities

### New Capabilities
- `conversation-file-store`: Per-conversation JSONL + sidecar file storage for durable message history, including the `ConversationMessageStore` interface, file/legacy resolver, write-queue concurrency, and deletion lifecycle hooks.
- `raw-message-debug-log`: File-based debug log replacing `model_raw_messages`, including retention/cleanup.

### Modified Capabilities
- `conversation`: Message storage backend changes from SQLite-only to file-or-legacy-SQLite depending on conversation age; append-only ordering and message-type guarantees are preserved, but the underlying persistence medium and id-generation scheme change.
- `conversation-pagination`: Pagination (`beforeMessageId`/`limit`) must be served correctly from both the file store (via sidecar-assisted seeks) and the legacy SQLite store.
- `conversation-compaction`: Compaction-anchor lookup (last `compaction_summary` message) and range-since-anchor reads move from SQL queries to file-store methods.
- `db-retention-job`: `RetentionJob` no longer deletes `model_raw_messages`/`stream_events` rows from SQLite; instead deletes/rotates the corresponding debug log files.
- `chat-session-retention`: the archived-chat-session hard-delete sweep additionally deletes the session's conversation file, sidecar, and any debug log files from disk alongside its existing cascade SQL deletes.
- `write-buffer`: `ConvMessageBuffer` is retargeted to flush against the new `ConversationMessageStore` instead of directly issuing SQL inserts; `RawMessageBuffer` is retargeted to flush against the new file-based debug log. The `stream_events` `WriteBuffer` usage is removed.
- `engine-stream-processor`: `StreamProcessor` no longer persists `stream_events`; the `onStreamEvent` live broadcast path (including `blockId`/`seq` enrichment, which the frontend still needs for grouping) is unchanged, but the DB-persistence side effect is deleted.
- `stream-event-enricher`: `StreamEventEnricher` is retained — the frontend still relies on `blockId`/`seq` for live block grouping — but its role is narrowed to enriching the in-memory broadcast event only; it no longer feeds a persistence `WriteBuffer`.
- `stream-reconnect-replay`: **Removed** — this capability's entire mechanism (the `conversations.getStreamEvents` RPC and its backing query) is deleted as it has no live caller.

## Impact

- **Affected code**: `src/bun/conversation/*` (messages.ts, conv-message-buffer.ts, context.ts, context-estimator.ts, cross-engine-context.ts, decision-context-injector.ts), `src/bun/workflow/session-memory.ts` (DI refactor + path fix), `src/bun/handlers/conversations.ts`, `src/bun/handlers/chat-sessions.ts`, `src/bun/handlers/tasks.ts`, `src/bun/workflow/tools/board-tool-executor.ts`, `src/bun/engine/execution/{chat,human-turn,code-review}-executor.ts`, `src/bun/engine/stream/stream-processor.ts`, `src/bun/engine/stream/raw-message-buffer.ts`, `src/bun/jobs/retention-job.ts`, a new `ConversationFileDeleter` collaborator, `src/bun/db/stream-events.ts` (deleted), `src/bun/db/migrations/*` (new migration to drop `stream_events`; existing tables/migrations relating to it become historical), `src/shared/rpc-types.ts` (remove `conversations.getStreamEvents` types).
- **APIs**: `conversations.getStreamEvents` RPC method removed (**BREAKING** for any caller — confirmed none exist in the frontend today).
- **Data**: New per-conversation files under `~/.railyn/conversations/`; new debug log files under `~/.railyn/debug/` (or similar); `stream_events` table dropped; `model_raw_messages` table dropped once file-based logging is live; `conversation_messages` table remains (read-only) for pre-existing conversations indefinitely (no backfill).
- **No frontend behavior changes are expected** for durable message history (same REST/WS contract for `getMessages`/`onNewMessage`) or for live streaming (unchanged in-memory WS push path).
